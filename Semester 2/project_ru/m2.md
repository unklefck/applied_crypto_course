**Цель спринта:** Реализовать основные режимы конфиденциального шифрования (CBC, CFB, OFB, CTR).

| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |

|      | - Обновлённые примеры использования с новыми режимами и обработкой IV.                                            |           |
|      | - Инструкции по новым тестам совместимости с OpenSSL.                                                              |           |

#### **2. Парсер интерфейса командной строки (CLI)**

CLI должен быть расширен для поддержки новых режимов и параметра IV.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| CLI-1 | Аргумент `--mode` **должен** быть расширен и принимать новые значения: `cbc`, `cfb`, `ofb`, `ctr`.               | Must      |
| CLI-2 | Инструмент **должен** корректно обрабатывать IV в зависимости от операции:                                         | Must      |
|      | - **Для шифрования:** **должен** генерироваться криптографически стойкий случайный IV. Флаг `--iv` при шифровании **не должен приниматься**. |           |
|      | - **Для расшифрования:** флаг `--iv IV` **должен** поддерживаться. IV **должен** передаваться как шестнадцатеричная строка. |           |
| CLI-3 | CLI **должен** проверять, что `--iv` указывается только в режиме расшифрования. Если он задан при шифровании, его следует игнорировать с предупреждением либо считать это ошибкой. | Should |

**Примеры вызова:**
```bash
# Шифрование (IV генерируется, не передаётся)
$ cryptocore --algorithm aes --mode cbc --encrypt --key 000102...0f --input plaintext.txt --output ciphertext.bin

# Расшифрование (IV должен быть передан пользователем)
$ cryptocore --algorithm aes --mode cbc --decrypt --key 000102...0f --iv AABBCCDDEEFF00112233445566778899 --input ciphertext.bin --output decrypted.txt
```

#### **3. Базовая криптографическая реализация**

Четыре новых режима должны быть корректно реализованы с нуля на основе уже существующего примитива AES.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| CRY-1 | Все требования Спринта 1, относящиеся к использованию примитива AES (CRY-1, CRY-2), **должны** по-прежнему выполняться. | Must |
| CRY-2 | Логика режимов **CBC, CFB, OFB и CTR** **должна** быть реализована студентом с нуля.                              | Must      |
| CRY-3 | **Требования, специфичные для режимов:**                                                                           | Must      |
|      | - **CBC (Cipher Block Chaining):** должен быть реализован механизм сцепления, при котором каждый блок шифртекста XOR'ится со следующим блоком открытого текста. Требуется padding (PKCS#7). |           |
|      | - **CFB (Cipher Feedback):** должен быть реализован как потоковый шифр. Размер сегмента должен совпадать с полным размером блока (128 бит). Padding не требуется. |           |
|      | - **OFB (Output Feedback):** должен быть реализован как потоковый шифр с генерацией гаммы, независимой от открытого текста. Padding не требуется. |           |
|      | - **CTR (Counter):** должен быть реализован как потоковый шифр. Счётчик должен инициализироваться из IV и увеличиваться для каждого блока. Padding не требуется. |           |
| CRY-4 | **Логика padding должна быть обновлена:**                                                                          | Must      |
|      | - Режимы, которым нужен padding (ECB, CBC), **должны** продолжать использовать PKCS#7.                            |           |
|      | - Потоковые режимы, которым padding не нужен (CFB, OFB, CTR), **не должны** дополнять открытый текст. Реализация должна корректно обрабатывать неполный последний блок, не изменяя его размер. |           |

#### **4. Обработка вектора инициализации (IV)**

Безопасная и корректная работа с IV критична для безопасности этих режимов.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| IV-1 | **Для шифрования:**                                                                                                 | Must      |
|      | - IV **должен** генерироваться с помощью криптографически стойкого генератора случайных чисел (CSPRNG).           |           |
|      | - **Для Python:** использовать `os.urandom(16)`.                                                                    |           |
|      | - **Для C:** использовать `RAND_bytes()` из OpenSSL.                                                                |           |
| IV-2 | Сгенерированный IV **должен** записываться в начало выходного файла с шифртекстом.                                 | Must      |
|      | - **Формат файла при шифровании:** `<16-byte IV><Ciphertext Bytes>`                                                |           |
| IV-3 | **Для расшифрования:**                                                                                              | Must      |
|      | - Если передан аргумент `--iv`, инструмент **должен** использовать это значение для расшифрования.                 |           |
|      | - Если аргумент `--iv` **не** передан, инструмент **должен** прочитать первые 16 байт входного файла шифртекста и использовать их как IV. Оставшиеся байты являются собственно шифртекстом для расшифрования. |           |
| IV-4 | IV **должен** всегда иметь длину 16 байт (размер блока AES) для всех режимов.                                     | Must      |

#### **5. Файловый ввод-вывод**

Логика работы с файлами должна быть обновлена для поддержки нового формата с IV в файле.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| IO-1 | **При шифровании:** инструмент **должен** записывать сгенерированный 16-байтный IV в выходной файл до записи любых байтов шифртекста. | Must |
| IO-2 | **При расшифровании (без `--iv`):** инструмент **должен** прочитать первые 16 байт входного файла для получения IV. Далее расшифрование должно выполняться по оставшейся части файла. | Must |
| IO-3 | Если входной файл для расшифрования слишком короткий и не содержит полного IV (менее 16 байт), инструмент **должен** выдавать понятную ошибку. | Must |

#### **6. Тестирование и верификация**

Совместимость со стандартным инструментом OpenSSL является ключевым требованием этого спринта.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| TEST-1 | **Тест полного цикла:** шифрование и последующее расшифрование файла одним и тем же инструментом **должно** всегда возвращать файл, идентичный исходному, для каждого нового режима. | Must |
| TEST-2 | **Тест совместимости — ваш инструмент -> OpenSSL:**                                                                | Must      |
|      | 1. Зашифровать файл с помощью вашего инструмента (например, в режиме CBC).                                         |           |
|      | 2. Расшифровать полученный файл шифртекста с помощью CLI-команды OpenSSL.                                         |           |
|      | 3. Расшифрованный OpenSSL файл **должен** совпасть с исходным открытым текстом.                                   |           |
|      | `openssl enc -aes-128-<mode> -d -K <key_hex> -iv <iv_hex> -in <ciphertext_file> -out <decrypted_file>`           |           |
| TEST-3 | **Тест совместимости — OpenSSL -> ваш инструмент:**                                                                | Must      |
|      | 1. Зашифровать файл с помощью CLI-команды OpenSSL.                                                                  |           |
|      | 2. Расшифровать полученный файл шифртекста с помощью вашего инструмента. Нужно передать корректный IV, использованный при шифровании. |           |
|      | 3. Расшифрованный вашим инструментом файл **должен** совпасть с исходным открытым текстом.                        |           |
|      | `openssl enc -aes-128-<mode> -K <key_hex> -iv <iv_hex> -in <plaintext_file> -out <ciphertext_file>`              |           |
| TEST-4 | Студент **должен** привести в `README.md` тестовый сценарий или подробные команды, демонстрирующие успешные тесты совместимости для каждого из четырёх новых режимов. | Should |

**Пример команд совместимости с OpenSSL:**
```bash
# 1. Шифрование вашим инструментом, расшифрование OpenSSL
$ cryptocore --algorithm aes --mode cbc --encrypt --key 000102...0f --input plain.txt --output cipher.bin
# Извлечь IV из первых 16 байт файла cipher.bin (например, через `dd` или `hexdump`)
$ dd if=cipher.bin of=iv.bin bs=16 count=1
$ dd if=cipher.bin of=ciphertext_only.bin bs=16 skip=1
$ openssl enc -aes-128-cbc -d -K 000102030405060708090A0B0C0D0E0F -iv $(xxd -p iv.bin | tr -d '\n') -in ciphertext_only.bin -out decrypted.txt

# 2. Шифрование OpenSSL, расшифрование вашим инструментом
$ openssl enc -aes-128-cbc -K 000102030405060708090A0B0C0D0E0F -iv AABBCCDDEEFF00112233445566778899 -in plain.txt -out openssl_cipher.bin
$ cryptocore --algorithm aes --mode cbc --decrypt --key 000102...0f --iv AABBCCDDEEFF00112233445566778899 --input openssl_cipher.bin --output decrypted.txt
```
