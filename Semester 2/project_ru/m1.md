### **Проект: CryptoCore — документ технических требований (Спринт 1)**

**Цель спринта:** Развернуть кодовую базу и реализовать базовые операции блочного шифра в режиме ECB.

#### **2. Парсер интерфейса командной строки (CLI)**

Инструмент должен принимать аргументы в определённом и предсказуемом формате.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| CLI-1 | После сборки инструмент **должен** запускаться из командной строки как `cryptocore`.                               | Must      |
| CLI-2 | CLI **должен** принимать следующие аргументы:                                                                       | Must      |
|      | - `--algorithm ALGORITHM` : указывает шифр. В этом спринте **должно** поддерживаться значение `aes`.              |           |
|      | - `--mode MODE` : указывает режим работы. В этом спринте **должно** поддерживаться значение `ecb`.                |           |
|      | - `--encrypt` или `--decrypt` : для указания операции **должен** быть задан **ровно один** из этих флагов.        |           |
|      | - `--key KEY` : **должен** принимать строку, представляющую ключ. Для AES-128 это **должно** быть значение длиной 16 байт. |           |
|      | - `--input INPUT_FILE` : **должен** принимать путь в файловой системе к входному файлу.                            |           |
|      | - `--output OUTPUT_FILE` : **должен** принимать путь в файловой системе к выходному файлу.                         |           |
| CLI-3 | Аргумент `--key` **должен** передаваться как шестнадцатеричная строка (например, `00112233445566778899aabbccddeeff`). | Must      |
| CLI-4 | CLI **должен** проверять корректность всех аргументов. Если какой-либо аргумент отсутствует, имеет неверный формат или конфликтует с другими (например, одновременно заданы `--encrypt` и `--decrypt`), инструмент **должен** вывести понятное сообщение об ошибке в `stderr` и завершиться с ненулевым кодом возврата. | Must |
| CLI-5 | Если аргумент `--output` не указан, инструмент **может** формировать имя выходного файла по умолчанию (например, `input.txt.enc` для шифрования и `ciphertext.enc.dec` для расшифрования). | Could |

**Примеры вызова:**
```bash
# Шифрование
$ cryptocore --algorithm aes --mode ecb --encrypt --key 000102030405060708090a0b0c0d0e0f --input plaintext.txt --output ciphertext.bin

# Расшифрование
$ cryptocore --algorithm aes --mode ecb --decrypt --key 000102030405060708090a0b0c0d0e0f --input ciphertext.bin --output decrypted.txt
```

#### **3. Базовая криптографическая реализация**

Реализация должна корректно использовать примитив AES для выполнения операций в режиме ECB.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| CRY-1 | Реализация **должна** использовать **AES-128** (128-битный блочный шифр с 128-битным ключом).                     | Must      |
| CRY-2 | Базовые функции шифрования и расшифрования AES (сам примитив) **не должны** реализовываться с нуля.               | Must      |
|      | - **Для Python:** **должен** использоваться модуль `AES` из `Crypto.Cipher` (библиотека `pycryptodome`).         |           |
|      | - **Для C:** **должны** использоваться функции `AES_encrypt` и `AES_decrypt` из `openssl/evp.h` или `openssl/aes.h`. |           |
| CRY-3 | Логика **режима ECB** (разбиение входа на блоки, дополнение, вызов примитива AES для каждого блока) **должна** быть реализована студентом. | Must |
| CRY-4 | Реализация **должна** обрабатывать **padding**. **Должен** использоваться стандарт PKCS#7.                        | Must      |
|      | - **При шифровании:** открытый текст **должен** дополняться до кратности размеру блока 16 байт.                  |           |
|      | - **При расшифровании:** padding **должен** проверяться и удаляться после расшифрования.                          |           |
| CRY-5 | Инструмент **должен** корректно работать как с текстовыми, так и с бинарными файлами. Вход и выход **должны** обрабатываться как бинарные потоки (например, режимы `rb` и `wb` в Python). | Must |

#### **4. Файловый ввод-вывод**

Инструмент должен надёжно читать данные из файловой системы и записывать их в неё.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| IO-1 | Инструмент **должен** читать всё содержимое файла, указанного через `--input`.                                     | Must      |
| IO-2 | Инструмент **должен** записывать результат (зашифрованные или расшифрованные данные) в файл, указанный через `--output`. | Must |
| IO-3 | Инструмент **должен** корректно обрабатывать файловые ошибки (например, если входной файл не существует или недоступен для чтения). Он должен вывести информативную ошибку в `stderr` и завершиться с ненулевым кодом возврата. | Must |

#### **5. Тестирование и верификация**

Поставляемый результат должен быть функционально корректным.

| ID  | Описание требования                                                                                                 | Приоритет |
| :-- | :------------------------------------------------------------------------------------------------------------------ | :-------- |
| TEST-1 | Основной приёмочный тест: шифрование файла с последующим расшифрованием **должно** давать файл, полностью идентичный исходному. | Must |
|      | Команда `$ diff original_file.txt decrypted_file.txt` не должна показывать различий.                               |           |
| TEST-2 | Студент **должен** предоставить простой тестовый сценарий или документированный пример в `README.md`, демонстрирующий этот тест полного цикла. | Should |
| TEST-3 | Зашифрованный текст, создаваемый инструментом, **следует** сверить с эталонной реализацией (например, с `openssl enc -aes-128-ecb` в командной строке), чтобы убедиться в корректности реализации ECB. | Should |

**Пример команды для проверки через OpenSSL:**
```bash
# Шифрование с OpenSSL для сравнения
$ openssl enc -aes-128-ecb \
    -K 000102030405060708090a0b0c0d0e0f \
    -in plaintext.txt \
    -out ciphertext_openssl.bin \
    -nopad # Внимание: используйте -nopad только если файл уже имеет длину, кратную 16 байтам. Иначе ваша реализация должна сама обрабатывать padding.
```

### **Сводка обязательных технических решений**

*   **Язык:** C или Python.
*   **Основная криптобиблиотека:**
    *   **Python:** `pycryptodome` (`pip install pycryptodome`)
    *   **C:** OpenSSL (`libcrypto`, линковка через `-lcrypto`)
*   **Стандарт дополнения:** PKCS#7.
*   **Формат ключа:** шестнадцатеричная строка.
*   **Режим работы:** Electronic Codebook (ECB).

Следование этим требованиям позволит студенту успешно подготовить структурированную, безопасную и работоспособную кодовую базу, соответствующую цели реализации базовых операций блочного шифра.
