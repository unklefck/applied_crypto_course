### **Проект: Создание "CryptoCore" — Минималистического Криптографического Провайдера**

**Общая цель:** Студенты создадут консольную утилиту и связанную библиотеку (на C или Python), предоставляющие криптографические функции. API должен быть вдохновлён упрощёнными версиями команд OpenSSL `enc` и `dgst`. Основное внимание уделяется корректной реализации и пониманию, а не производительности или совместимости.

---

### **Веха 1: Основа проекта и базовые блоки**

*   **Цель спринта:** Создать кодобазу и реализовать основные операции блочных шифров.
*   **Ключевые концепции:** Проектирование ПО, режим ECB, базовый разбор аргументов командной строки.
*   **Результаты:**
    1.  Well-structured репозиторий кода (например, на GitHub) с `README.md` и системой сборки (например, Makefile для C, `setup.py` для Python).
    2.  Консольная утилита, которую можно вызывать как `cryptocore --algorithm aes --mode ecb --encrypt --key <ключ> --input <файл>`.
    3.  Реализация шифрования и дешифрования AES-128 в **режиме ECB** с нуля (использование проверенной библиотеки для основных функций раундов AES, такой как `pycryptodome` для Python или `libcrypto` от OpenSSL для C, **обязательно** из соображений безопасности и экономии времени. Фокус на режимах, а не на примитиве).
    4.  Базовый ввод-вывод для чтения открытого текста и записи шифротекста (и наоборот).
*   **Тестирование:** Убедиться, что шифрование, а затем дешифрование файла возвращает исходный файл.

### **Веха 2: Режимы работы и обработка векторов инициализации (IV)**

*   **Цель спринта:** Реализовать основные режимы обеспечения конфиденциальности.
*   **Ключевые концепции:** Векторы инициализации (IV), различия между CBC, CFB, OFB, CTR.
*   **Результаты:**
    1.  Расширить CLI для поддержки `--mode cbc`, `--mode cfb`, `--mode ofb`, `--mode ctr`.
    2.  Реализовать каждый из этих режимов с нуля.
    3.  Корректная обработка IV: генерация криптографически стойкого случайного IV для шифрования и приём IV на вход для дешифрования. IV должен быть добавлен в начало файла с шифротекстом.
*   **Тестирование:** Проверить совместимость: зашифровать файл своей утилитой и успешно расшифровать его командой OpenSSL CLI (и наоборот) для каждого режима. `openssl enc -aes-128-cbc -d -K <ключ_в_hex> -iv <iv_в_hex> -in <файл>`

### **Веха 3: Криптографически стойкая генерация случайных чисел**

*   **Цель спринта:** Реализовать безопасный источник случайности для ключей и IV.
*   **Ключевые концепции:** CSPRNG, источники энтропии, `/dev/urandom`.
*   **Результаты:**
    1.  Выделенный модуль CSPRNG в вашей библиотеке.
    2.  Функция для генерации криптографически стойких случайных байт (напр., `generate_random_bytes(num_bytes)`).
    3.  Интегрировать это в вашу утилиту:
        *   Если `--key` не указан, сгенерировать случайный 16-байтный (128-битный) ключ и вывести его в терминал.
        *   IV для режимов, которые этого требуют, должен всегда генерироваться случайно при шифровании.
*   **Тестирование:** Сгенерировать 1000 ключей и провести базовые статистические тесты (напр., проверить отсутствие дубликатов ключей, что крайне маловероятно).

### **Веха 4: Хеширование и целостность данных**

*   **Цель спринта:** Добавить криптографические хеш-функции для проверки целостности данных.
*   **Ключевые концепции:** Односторонние функции, устойчивость к коллизиям, структура Меркля-Дамгарда.
*   **Результаты:**
    1.  Новая команда CLI `cryptocore dgst --algorithm sha256 --input <файл>` для вычисления дайджестов.
    2.  Реализация SHA-256 с нуля (значительная, но очень образовательная задача). Альтернативно, *только для этой функции* можно использовать проверенную библиотеку, если фокус курса на другом.
    3.  Поддержка как минимум двух алгоритмов (напр., SHA-256, SHA3-256 или BLAKE2).
*   **Тестирование:** Захешировать известный файл и убедиться, что результат совпадает с результатом системной команды `sha256sum`.

### **Веха 5: Коды аутентичности сообщений (MAC)**

*   **Цель спринта:** Реализовать функции для обеспечения аутентичности и целостности данных.
*   **Ключевые концепции:** Симметричная аутентификация, конструкция HMAC.
*   **Результаты:**
    1.  Расширить команду `dgst` для поддержки `--hmac --key <ключ>`.
    2.  Реализовать HMAC с нуля, используя хеш-функцию SHA-256 из предыдущей вехи.
    3.  *(Бонус)* Реализовать AES-CMAC.
*   **Тестирование:** Сгенерировать HMAC для файла с ключом. Проверить его тем же ключом. Изменить один бит в файле или ключе и убедиться, что проверка не проходит.

### **Веха 6: Аутентифицированное шифрование**

*   **Цель спринта:** Объединить шифрование и аутентификацию в одну безопасную операцию.
*   **Ключевые концепции:** AEAD, одноразовый номер (nonce), ассоциированные данные (AAD), шифрование затем MAC vs. MAC затем шифрование.
*   **Результаты:**
    1.  Реализовать парадигму Encrypt-then-MAC как составную функцию.
    2.  Реализовать режим GCM с нуля (сложная, но полезная задача). Это включает реализацию умножения в поле Галуа.
    3.  Добавить новый режим `--mode gcm` и `--aad <данные>` в вашу команду шифрования CLI.
*   **Тестирование:** Зашифровать с AAD, затем расшифровать с правильным AAD. Попытаться расшифровать с неправильным AAD или изменённым шифротекстом и убедиться, что это завершается неудачей (т.е., не выводится расшифрованный открытый текст).

### **Веха 7: Формирование ключей и основа протоколов**

*   **Цель спринта:** Реализовать функции для безопасного формирования ключей из паролей и других ключей.
*   **Ключевые концепции:** Растяжение ключа, соль, счётчик итераций.
*   **Результаты:**
    1.  Новая команда CLI `cryptocore derive --password <пароль> --salt <соль> --iterations 100000 --length 32`.
    2.  Реализация PBKDF2-HMAC-SHA256 с нуля.
    3.  Функция для формирования иерархии ключей из мастер-ключа (напр., `derive_key(master_key, context, length)` с использованием HMAC детерминированным способом).
*   **Тестирование:** Сформировать ключ с заданным паролем, солью и счётчиком итераций. Убедиться, что он совпадает с результатом проверенной реализации (напр., `openssl kdf` от OpenSSL).

### **Веха 8: Финальная интеграция, тестирование и документация**

*   **Цель спринта:** Привести библиотеку в готовый вид, обеспечить надёжность и создать исчерпывающую документацию. Подготовиться к демонстрации.
*   **Ключевые концепции:** Тестирование ПО, документация, проектирование API.
*   **Результаты:**
    1.  Всеобъемлющий набор тестов, который проверяет каждую функцию на соответствие тестовым векторам NIST или из других известных источников.
    2.  Подробный документ `API.md`, объясняющий каждую публичную функцию в библиотеке.
    3.  `USERGUIDE.md` с примерами для каждой команды CLI.
    4.  Финальная презентация или демонстрационное видео, показывающее, как утилита шифрует, аутентифицирует и расшифровывает файл, а также показывает сбои проверки при изменении данных.
    5.  *(Бонус)* Реализовать пример простого "криточата", который использует вашу библиотеку для обеспечения конфиденциальности и целостности сообщений.
